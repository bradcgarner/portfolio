This Site

This is a temporary area where I'm keeping notes about development of this site (for myself, and for sharing).

Key:

  • General Description / Accomplishment
  • Learning Point
  • Not Happy With / Problem


So Far:

  • Used what I already learned about RWD, and recycled a 12-column exercise for my RWD grid
    • Updated that grid to allow 2-column thru 12-column (no 1, no 11)
    • Migrated all CSS for the grid to Sass
    • I think I have this in a format where I could easily recycle it.
    • I'd be interested in writing the HTML via JS. E.g. using very simple indents or minimal number of characters, writing the HTML with the appropriate element nesting and classes. I could easily script FileMaker to do that. I wonder if there are widgets out there that already do this, with a few parameters entered like class names?
    • The class customCol and customRow are just placeholder classes for now. Those are to indicate that this is where a custom class would go, if the column or row would be styled specifically, in addition to the columns-float-grid.scss styling.
    • The box classes are clearly placeholders, and are no indication of the final content. Only indication that the grid is working.
  • Started with a menu from a prior HTML5 class
    • Updated/expanded the menu
    • Added additional classes to the menu for greater flexibility in manipulating via JS and CSS (originally, the menu had no classes, and any manipulation had to be done via multi-tier parent>child relationships
    • Migrated styling of the menu to Sass
    • Learned the long and hard way about when to use & with &:hover and &::before, etc. Similarly had to remove a lot of > from the menu when I migrated to Sass.
    • Had a LOT of challenges getting the menu to go away after clicking (since this is a single-page site). Ended up scripting this as on-click, change 1st tier of menu to a display:none class. Then when we want to display again, there is another script that actives on-hover over the parent ul. The scripting I did seems to be fast, but seems roundabout. Suggestions?
    • I spent a huge amount of time trying to get all the menu lis to hover consistently (some are scripted onclick, some are anchor tags). I finally wrapped the anchor tags around lis and voilla! It works. But is there a better way?
    • Learned that one of the first things to check with positioning is that all parents are positioned. (I made the mistake of skipping a few childen at one point, and then things get weird.)
  • Scripted non-link menu actions to allow a single-page site
    • Strengthened positioning skills
    • Tried to get all divs to animate in from the left, always. But now I actually like how the previously-viewed divs come in from the right
    • I added a timeout in JS to allow the left-to-right script to complete before moving the section back to the left, so that scrolling is ALWAYS left-to-right. Only exception: clicking in rapid-fire it will return from the right (which I think is just fine). Better ideas?
    • Learned that animation can be done by changing classes and having a destination point only. Was trying to do this with @keyframes, which required me to know where the div was (left or right) then move it. Turned out to be way too complicated, and not necessary.
    • ...I am guessing I'll look back at this and laugh after learning React...
    • Learned a lot about using classes to affect display properties. Tried several animation techniques: learned that there are many, and simple sometimes is best.
  • Installed slicknav to use for the mobile menu.
    • Styled slicknav.scss to match my main.scss file. This turned out to be pretty easy.
    • Had a LOT of challenges getting the mobile menu to collapse upon click (which I assume is because this is a single-page site). Searched online, and found several people with the same problem, but NONE of the dozen or so suggestions was helpful. Ended up manually debugging this by clicking the mobile menu multiple times to figure out how slicknav was doing it. Basically slicknav is changing classes (similar to what I was doing).
    • I tried understanding whether there was a script already in slicknav that I could run to do this. No luck.
    • I was unsuccessful at getting the ENTIRE slicknav menu to collapse, but I can live with that for now.
    • Learned a lot about using classes to affect display properties.
  • Styled the banner
    • Within header>h1, explored letter-spacing and transform: scale. (I like wide header fonts, though this is not exactly Frank Lloyd Wright).
    • I had to wrap header>h1 in a div that I gave class headerBanner with property overflow:none. Without this, the h1 was spilling off to the right. I could not figure out why, and I tried to debug that a lot. Adding overflow:none to the header as a whole, means the menu doesn't work (it "needs" to overflow), and adding it to the body seems to prevent the body from scrolling (as if the body is limited to the dimensions of size: cover). Suggestions?? This one really bugged me.
